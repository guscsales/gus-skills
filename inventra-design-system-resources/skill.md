## DS Learning Procedure (MANDATORY before designing)

Every time you need to create or modify a design in Paper, run this procedure for each component type you will use. This is non-negotiable — it ensures pixel-perfect DS compliance.

### Step 1: Identify components needed

List every component type your design will use (e.g., "I need: Input, Button, Switch, Card, Alert").

### Step 2: For each component, deep-read the DS artboard

Use this exact sequence of Paper MCP calls:

```
1. get_children(artboardId)          — map top-level sections (Light / Dark)
2. get_children(lightSectionId)      — map component groups
3. get_children(componentGroupId)    — find the specific component variants
4. get_children(variantId)           — find the frame + label + annotation
5. get_computed_styles([frameIds])   — extract EXACT CSS: border-radius, padding, border-color, background, gap, etc.
6. get_computed_styles([textIds])    — extract EXACT typography: font-family, font-size, line-height, font-weight, color
```

**NEVER guess values from screenshots.** Always use `get_computed_styles` for the ground truth.

### Step 3: Build your component token map

Before writing any HTML, compile a token map like this:

```
COMPONENT: Input (md)
- border-radius: 8px
- padding: 8px 12px
- border: 1px solid #E1E4E8
- background: #FFFFFF
- text: Geist Regular 13/16 #18191B
- placeholder: Geist Regular 13/16 #6B7280

COMPONENT: Label
- font: Geist Medium 12/16 #18191B
- gap to input: 6px

COMPONENT: Helper text
- font: Geist Regular 11/14 #6B7280
```

### Step 4: Use the token map when writing HTML

Every `write_html` call must use values from your token map. Never hardcode from memory.

### Step 5: Verify after writing

After creating components, use `get_computed_styles` on your created nodes to verify they match the DS.

## Important Rules

1. **Always use `get_computed_styles`, never guess from screenshots** — screenshots lose precision
2. **The Light theme section is always the first child** of the artboard content area
3. **Each artboard has Light + Dark** — use Light for default designs unless dark mode is explicitly requested
4. **When a component doesn't exist in the DS**, compose it from existing primitives (Input + Label + Helper pattern)
5. **If the DS has been updated** (new artboards added), re-run `get_basic_info` on page `9-0` to discover new artboards
